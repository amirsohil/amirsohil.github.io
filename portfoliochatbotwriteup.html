Ask Amir — Building an AI Portfolio Assistant

Essay · AI Engineering · Prompt Design · Zero-Cost Infrastructure

The Problem

Static Portfolios Are a One-Way Broadcast

I've been on the receiving end of this enough times to know how it plays out. A recruiter or potential client lands on your portfolio. They have a specific question — does he work with Databricks? Has he done anything in finance? Is he actually available? — and the answer is somewhere on the page, buried in a bullet point they may or may not scroll to. Most don't scroll far. They move on.

A contact form doesn't fix this. Neither does a Calendly link. What fixes it is something that can answer the question immediately, accurately, and at any time — without me being online. I wanted a conversational interface that knew my full background and could speak on my behalf. Not a chatbot template dropped onto a page. Something built to match the site, built to answer well, and built in a way that demonstrated the same kind of thinking I claim professionally.

There was a second reason I wanted to build it from scratch rather than use an existing tool. Every technical decision in this project — model selection, knowledge base architecture, API security, token management — is the kind of decision I make for clients. Building this openly, and writing about it, is its own proof of work.

The goal wasn't to build something that looked impressive. It was to build something that worked — and to build it in a way that was worth writing about.

Architecture

How It Works

The system has three components: a static HTML frontend embedded in my portfolio, a serverless proxy running on Cloudflare Workers, and Groq's inference API running Llama 3.3 70B. The frontend and proxy are permanently free. The inference API has a generous free tier — 1,000 requests and 100,000 tokens per day — which is more than sufficient for portfolio traffic.

Browser ──────► Cloudflare Worker ──────► Groq API (Llama 3.3 70B)
                                  ◄────── Response ◄──────

No backend server. No database. No auth layer.
API key stored as encrypted Worker secret.

The frontend sends a POST request to the Worker containing the full conversation history and the user's latest message. The Worker holds the Groq API key as an encrypted environment variable — it never touches client-side code — authenticates with Groq, and forwards the payload. The response comes back in under two seconds in the vast majority of cases.

The entire frontend is vanilla HTML and JavaScript. No React, no build step, no dependency management. Everything — structure, styles, and logic — lives in a single file. When I want to update the knowledge base, I edit a text block and upload one file to GitHub. That simplicity is a deliberate choice, not a limitation.


The Hard Part

Writing the Brief

The model has no memory of who I am. Every conversation starts from zero. The only context it has is what I send — a structured text block in the system instruction before every request. I call it the knowledge base, and I underestimated how much work it would take to get it right.

A knowledge base is not a CV. A CV lists what you did. A knowledge base has to anticipate what someone might ask and give the model enough grounding to answer confidently, specifically, and honestly — without inventing credentials that don't exist. I hit both failure modes before I got there. Too sparse, and the model hedges constantly: "I believe Amir may have experience with..." — which reads as uncertainty about your own background, exactly the opposite of what you want. Too dense, and it buries the important detail or loses the thread on which project belongs to which period.

The structure I settled on mirrors how a well-briefed colleague would describe me: identity and availability first, then role by role with specific outcomes and tool references, then certifications and education. Each role includes the actual numbers — 200+ reports, 75% cycle reduction, 7+ workspaces. The model speaks in the first person throughout and is explicitly told to be specific rather than general.

Two additional rules took iteration to get right. The first is a scope guardrail: if asked anything outside my professional background — general knowledge, maths, coding help, current events — the assistant politely declines and redirects. Without this, the model will happily answer any question it can, which looks impressive in isolation but undermines the trust signal the tool is supposed to create. A portfolio assistant that doubles as a general-purpose chatbot raises a question about what it's actually doing with visitor data.

The second is a company inquiry rule: if a recruiter or hiring manager asks about a specific role or company, the assistant confirms openness and relevant skills, then directs them to email — it never fabricates details about companies not in its knowledge base. I added this after seeing a model confidently describe a company's product offering from thin air. That kind of hallucination in a professional context is worse than saying nothing.

The projects layer — KB_PROJECTS — required the same discipline applied specifically to project descriptions. Each entry includes the technology stack, scale metrics where meaningful (276,075 similarity edges, 32,581 loans), competition outcomes, and live URLs where they exist. The full projects list now covers over a dozen pieces of work across ML, graph databases, geospatial analysis, data storytelling, and BI — a breadth that would make the core KB too heavy to send on every request, which is why the two-layer split exists.

The hard part was not the code. It was writing a brief precise enough that the model could reason from it without inventing things I haven't done — and constrained enough that it wouldn't go off-script when someone tried to use it as a general assistant.

Architecture Decisions

Every Choice, and Why

Click any card to expand the reasoning.

Inference · Provider

Groq over Gemini, OpenAI, or self-hosted

I started with Gemini. It didn't survive contact with reality.

The first version of this was built on Google Gemini. The integration itself worked — responses came back, quality was reasonable, and the setup was quick. What I didn't anticipate was Google's new-project access controls. During development, my Cloud project was automatically flagged and access-denied with a "please contact support" error. It took over 24 hours to resolve. A second project I created hit the same wall immediately after. A tool that silently breaks because of an upstream policy decision — with no warning, no ETA, and no clear resolution path — is not something I'm willing to put on a permanently live portfolio page.

OpenAI requires a credit card even for free-tier access, which introduces billing exposure on a publicly accessible endpoint. Self-hosted open-source models require infrastructure I have no interest in maintaining for a static portfolio page. Groq requires only an email address, activates immediately, has no project-blocking behaviour, and runs high-quality open-source models at fast inference speeds. The free tier resets daily and has been completely stable throughout.

Outcome: Immediate setup, no billing risk, consistent sub-2-second responses. No incidents since switching.
Model · Selection

Llama 3.3 70B Versatile

Best balance of response quality and daily token budget on Groq's free tier.

Groq's free tier offers several models. The choice came down to two candidates: llama-3.3-70b-versatile (100K tokens/day) and meta-llama/llama-4-scout-17b-16e-instruct (500K tokens/day).

Scout has five times the daily token budget, which matters because the knowledge base is large and gets sent with every request. However, at 17 billion parameters it is a smaller model than the 70B Versatile, and the quality difference is noticeable on nuanced or ambiguous questions — exactly the kind a recruiter might ask. I opted for quality over headroom.

The fallback strategy handles the limit anyway: the Cloudflare Worker tries llama-3.3-70b-versatile first, and automatically retries with llama-4-scout if a rate limit is returned. The user never sees an error.

Outcome: Best quality primary model, automatic fallback if daily limit is reached.
Security · API Key

Cloudflare Worker as proxy

The API key never appears in client-side code.

Calling an LLM API directly from the browser requires the API key to be present in the JavaScript — which means anyone who opens the page source can read it, copy it, and use it to exhaust your quota or rack up charges on a paid plan.

A Cloudflare Worker is a serverless function that runs at the edge. It accepts requests from the browser, retrieves the API key from an encrypted environment variable that only the Worker can read, and forwards the request to Groq. The key is never transmitted to the client. Cloudflare's free tier allows 100,000 Worker requests per day — sufficient for any reasonable portfolio traffic.

This is the standard pattern for protecting API keys in frontend-only applications, and it adds no meaningful latency since Cloudflare's edge network is geographically distributed.

Outcome: API key fully protected, zero added infrastructure cost.
Frontend · Framework

Vanilla HTML and JavaScript, no frameworks

The simplest thing that works. No build step, no dependencies, no bundle size.

My portfolio is a static site. Adding a React or Vue component would require a build pipeline, a separate deployment process, and either an iframe or a significant restructuring of the existing site. None of that was warranted for a single-page chat interface.

Vanilla JavaScript handles the DOM manipulation, message rendering, and API calls cleanly. The entire chatbot — HTML structure, CSS styling, and all logic — lives in one file. Updating the knowledge base is a text edit. Deploying an update is uploading one file to GitHub.

The constraint also forced discipline. Without a component library, every design decision was intentional — which produced a cleaner result than reaching for pre-built components would have.

Outcome: Single-file deployment, zero dependencies, full design control.
Design · Integration

Native to the portfolio, not a dropped-in widget

Recruiters notice when something doesn't belong. The chatbot had to feel like part of the site.

Generic chat widget UIs — floating buttons, rounded white bubbles, off-brand colour schemes — signal that a tool was added as an afterthought. They also signal a specific kind of technical decision: I found something ready-made and dropped it in. That is a meaningful signal on a portfolio that is supposed to demonstrate engineering judgement.

The chatbot uses the same Raleway headings and Source Sans Pro body text as the rest of the site. The accent colour is the same #CDFD50 lime green that appears in headers, links, and section markers across all portfolio pages. The dark background, border treatment, and button style follow the existing design language exactly.

Outcome: The chatbot feels native, not bolted on. The design decision is part of the demonstration.
UX · Suggestions

Suggestion chips on load

Most visitors won't know what to ask. The chips remove the blank-input problem.

A blank text input is a conversion killer. If someone arrives at the chatbot without a specific question already in mind, the empty placeholder text — no matter how well-written — creates a moment of friction. They stall, then leave.

Five suggestion chips appear on load, each representing a question a recruiter or client is genuinely likely to ask: Power BI experience, the HR SaaS product, ML projects, remote availability, and awards. One tap sends the question. The chips disappear after the first message to clear space for the conversation.

Outcome: Lower friction entry, higher engagement rate, guided conversation without feeling scripted.
Prompt Design · Guardrails

Scope rule and company inquiry rule

A portfolio assistant that answers anything undermines the very trust signal it's supposed to create.

Without explicit guardrails, an LLM will answer any question it can — general knowledge, maths, coding problems, current events. That might look capable in isolation, but it creates a trust problem on a portfolio page. A visitor wondering "what is this thing actually doing with my input?" is not a visitor who books a session or sends an email.

The scope rule is simple: if a question falls outside my professional background — experience, skills, projects, education, availability — the assistant politely declines and redirects. This isn't a technical limitation, it's an intentional design decision. The assistant is a specialist, not a general chatbot, and that specialisation is part of the signal.

The company inquiry rule addresses a specific hallucination risk: if a recruiter asks about a role at a company the assistant doesn't know about, it will sometimes confidently describe the company's products, team size, or culture from its training data — information that may be outdated, wrong, or fabricated. The rule redirects all company-specific enquiries to email, where a real conversation can happen with accurate information.

Outcome: Consistent, trustworthy behaviour. The assistant does one thing well rather than many things unreliably.


Retrospective

What I Would Do Differently

Streaming responses. Right now the chatbot waits for the complete reply before rendering anything — the typing indicator runs, then the full response appears at once. Groq supports streaming via server-sent events, which would let text appear incrementally as it's generated. The perceived latency difference is significant even when actual response time is identical. This is the highest-priority improvement I haven't made yet, mainly because it requires a more complex Worker implementation — streaming a chunked response rather than a JSON blob — and I haven't had a reason to prioritise it over other things.

Conversation analytics. I wrote the knowledge base based on what I assumed recruiters and hiring managers would ask. I have no way of knowing whether those assumptions are right. Anonymised logging — a Cloudflare Worker webhook writing to a Google Sheet, or a D1 database — would show me which questions actually get asked, which ones get a poor response, and what gaps exist in the knowledge base. Without that feedback loop, I'm iterating blind.

Conversation memory across sessions. Every page visit starts fresh. Someone who comes back the next day to follow up has to re-establish context from the beginning. Persisting the last conversation in localStorage is a small change that would make the experience noticeably more coherent for returning visitors — and returning visitors are exactly the people worth impressing most.

Smarter project keyword detection. The current needsProjects() function scans the latest message against a keyword list. It handles direct project names and explicit intent phrases well, but misses natural phrasing that doesn't match any keyword — "can I see something you've shipped?" or "show me what you've done" won't trigger injection even though they clearly should. A broader and more carefully maintained keyword list, or a lightweight intent classifier, would close that gap without meaningfully affecting token cost.
